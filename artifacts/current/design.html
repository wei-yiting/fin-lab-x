§4

Fallback chain 從 corpus probe 反推，不從假設正推

原本的假設

block heading 寫在 ALL-CAPS line（看起來符合 SEC 公司行文慣例）。設計 detection 規則用 isupper() + length filter。

實際 probe 結果

24 個跨公司 probe 跑下來只有 8% 真的成立。多數公司用 Title-Case sub-heading、有些落在 markdown H3、有些落在 H4。原 rule 套不上。重新做 30 個 case 的跨 sector probe，發現必須 H3 prefer → H4 fallback → text-based fallback 三層才能涵蓋全部。Fallback chain 的形狀不是先驗推出來的，是看到 probe 失敗模式反推的。

帶走的 parser / detector 不要先寫規則再去找對應案例 — 這方向會 over-fit 你最初的 assumption。先跑跨 corpus probe 看實際分布，再讓 fallback chain 從失敗模式反推。Probe 要跨 sector，分布偏的 corpus 會騙過你。
§5

Prelude 用 payload metadata，不切成獨立 chunk

兩個方案

(A) prelude 像普通段落 chunk + embed，retrieve 時若 hit 到 block chunk，再額外 Qdrant query 拉回該 Item 的 prelude chunks 拼進 context。
(B) prelude 整段塞進每個 block chunk 的 payload metadata，retrieve 時直接讀。

為什麼選 (B)

(A) 三層複雜性堆疊 — retrieve-time 多一次 Qdrant query (N+1)、prelude embedding 跟 block embedding 在 vector space 競爭 ranking slot、retrieve 端要靠 block_heading=None flag 區別 prelude 跟 block 內容。
(B) 只多付 ingest-time storage（prelude 重複 per block chunk，跨 corpus 估計 100-300 MB 可接受），retrieve 端零額外邏輯。

帶走的 cross-cutting context（item-level scoping、章節 disclaimer、metadata）— ingest-time 預付 storage 通常贏 retrieve-time 動態組裝。Storage 是 commodity，retrieve-time 每多一次 query / 每多一條 if 都是 correctness 風險面。
§6

Vector DB payload 該 self-contained，別用 SQL row 的思維

schema 長相

每個 chunk 在 Qdrant 的 payload 不只有 chunk text，還塞 ticker / year / filing_date / item / block_heading / prelude 等所有 LLM 需要的 context — 整段重複 per chunk。看起來「不夠正規化」：每個 filing 的 metadata 在所屬的所有 chunks 裡都重複一次，prelude 100-300 MB storage overhead。

為什麼 vs 看似乾淨的 normalized schema

Vector DB 不是 SQL，payload 是 document 不是 row。要在 retrieve-time 補拼 context（先 vector match、再從別處 lookup 補 metadata 跟 prelude）就需要 N+1 query，每次 retrieve latency 直接翻倍 — 而 retrieve-time latency 是 user-facing。Storage 是 commodity，retrieve-time 每多一次 query 都是 correctness 風險面跟延遲面。

帶走的 設計 vector DB schema 時放棄 SQL「DRY normalize」思維。每個 payload 是 self-contained document，retrieve 端 zero lookup 直接給 LLM 用。Denormalization 在 vector DB 是常態不是 anti-pattern — 為了 chunk text 不重複而設計的 normalized schema，反而拖累 RAG latency 跟 correctness。
§7

不同階段的 cache 該分層，invalidation lifecycle 不同

實際 setup

filing store 存 ParsedFiling JSON、Qdrant 存 chunk embeddings — 一條 RAG pipeline 兩個 cache 同時存在。看起來重複 / 浪費 storage。

為什麼分層 vs 只留 Qdrant

兩 cache 的 invalidation trigger 完全不同 — filing store 在 SEC filing 改版或 edgartools 升級時失效（罕見、年級事件）；Qdrant 在 chunking 策略改 / embedding model 換時失效（中等頻率、調 RAG 時就會發生）。如果只有 Qdrant，想跑「chunk_size 512 vs 1024 對照 eval」就要重做 fetch + parse + chunk + embed 整段（SEC HTTP rate limit、慢、quota 風險），但 fetch+parse 階段其實沒改。分層後，改下游只重做下游、上游 cache 仍有效。

帶走的 ML pipeline 設計 cache 時不要全部塞同一層。識別每階段的 axis of change — 什麼會改變、什麼不會。把 cache 分層，invalidation 影響範圍 scope 到該層。改 chunk 策略時只重做 chunk 後段、改 model 時只重做 embed 後段、上游 fetch / parse cache 仍能 reuse。否則每次小改動都觸發整 pipeline 重跑，experiment iteration 速度被卡住。
§8

Sibling rename via git mv 保留 A/B baseline

三條 migration 路線

A in-place rewrite — 直接覆寫舊 module。
B dual contract — 同 module 內保留兩 API 共存。
C sibling rename — git mv 舊版到 _html suffix，新版用主名。我一開始傾向 A，看起來最乾淨。

為什麼換到 C

你點出關鍵 — 要證明新版真的更好，需要跑 A/B eval 對照 baseline。換完之後沒辦法跑舊版 = 失去 baseline。B 維護兩 API contract 成本太高。C 兩 module 各自獨立演化，A/B eval 一邊打舊 collection、一邊打新 collection 直接 metric 對照；sunset PR 是一次 rm -rf 結束。

帶走的 Migration 不只是「替換」，是「替換 + 證明替換值得」 — 證明需要 baseline 還能跑。Sibling rename via git mv 是保留 baseline 不污染主路徑的乾淨手法 — 多一道工序，換到 unbiased evaluation 能力。
§10

Pipeline 每個階段邊界都該有 inspect artifact

實際設計

RAG pipeline 多階段 — fetch → detect → chunk → embed → retrieve。出 bug 時症狀通常只在最後一階段顯露（「LLM 答錯」），但 bug 可能在任何一階段。設計 inspect subcommand：把 ParsedFiling 渲成 human-readable markdown 寫到 inspect/{ticker}_{year}.md，整份 10-K 的 item / prelude / block heading / block content 全部顯式展開。

為什麼一定要這層 artifact

沒這個 artifact 時，debug「LLM 答錯」要同時調查所有階段。有了它，先把 inspect markdown 跟 SEC 原文 side-by-side diff — Item list 對嗎？prelude 範圍對嗎？block heading 是真 heading 還是 ToC artifact？對了 → bug 不在 detection，去 chunking / embed / retrieve 找；錯了 → 修 detection 不用碰下游。「retrieval 出問題不出在這裡」變成人工可確認的命題。

帶走的 多階段 ML pipeline 在每個階段邊界都該產 inspect artifact — 不需 production 必要，debug 用。格式要讓人類肉眼能對照 ground truth（markdown / table / 圖表都比 JSON 好）。沒 inspect artifact 時 debug 變成同時拆所有 stage；有了之後 debug 變成 binary search 找出錯 stage。一次性投資、長期 debug 加速。
§15

多層 anchoring + 四象限 failure mining

提案的演化

retrieval quality 怎麼評？我一開始提 item-level anchoring — retrieved chunk 來自哪個 Item 算 hit。你點出太粗：Item 1A 是 risk factors 整個 50 頁，retrieve 到對 Item 不代表 retrieve 到對段落。加上 snippet-level（sentence 50-200 char）substring 比對。

為什麼兩層比一層強

兩個獨立 hit/miss 訊號交叉 → 四象限：

  • snippet hit + item hit — 完美
  • snippet miss + item hit — 對章節錯段落，chunking quality 問題
  • snippet miss + item miss — detection / corpus 問題
  • snippet hit + item miss — 邏輯不可能

象限分布直接告訴你 bug 在 pipeline 哪一層。

帶走的 AI 系統 eval 別只有單一 metric — 多層 anchoring 是把「答錯了」分解成「哪一層 component 答錯」的關鍵。Recall@k 告訴你「失敗」、象限分布告訴你「哪個 stage 失敗」 — 這兩個訊號驅動的 fix loop 完全不一樣。
§15

Sunset trigger 用 deliverable，不用 metric threshold

選擇點

A/B 跑完什麼算結束？兩條路 — pre-commit「新版 recall@5 必須比 baseline 高 X%」（metric threshold），或「跑完 + 寫完 retrospective doc 歸納 failure mode」（deliverable）。

為什麼選 deliverable

memory feedback 直接禁止 pre-set outcome threshold。實質理由 — metric threshold 會反向施壓：如果新版跑出來是「乾淨但 metric 持平」，threshold 會逼你 fudge 直到達標。deliverable 把「歸納了 N 個 failure mode」當完成訊號，比較誠實面對 experiment 真實結論（包括「improvement 不在 metric，在 maintainability」這種結論）。

帶走的 Exploratory experiment 的 acceptance criteria 用 deliverable（「寫完 retrospective」「列出 N 個 root cause」），不用 metric threshold（「必須 beat baseline X%」）。後者把 experiment 變成自證題；前者讓 experiment 真的回答 design 問題。
§15

One-shot experiment data 跟 production eval 隔離

當下決定

A/B 的 50 rows dataset 要塞進既有 sec_retrieval production scenario（加 --collection flag），還是另開 sibling scenario directory？我選了塞進去 — 看起來「省一個 file」。memory feedback 跳出禁止這樣做。

實際代價

production scenario 長期演化、experiment scenario 一次性。混在一起後 — dataset row semantic 來源不一致（experiment 行強調 detection path coverage、production 行強調 user-facing query 廣度）；experiment 結束時 git diff 上分不出哪幾 row 是 experiment 的；scorer 為了 --collection 加的 logic 變成永久 dead code。改成 sibling directory 後，sunset 就是 rm -rf 整 directory，乾淨俐落。

帶走的 Time-bounded experiment 永遠開自己的 scenario directory。混進 production eval 看起來「省一個 file」，實際是把 sunset cleanup 從「rm dir」變成「猜哪些 row 該砍」。Schema 看似一樣不代表 lifecycle 一樣 — container 要分。

背景與動機

Architecture

FinLab-X v2 RAG pipeline 仰賴 ~1000 行自製 HTML parsing（preprocessor / heading promoter / converter / cleaner / downloader 共五個 module），每家新公司可能引入 HTML quirk（JNJ / MSFT / BAC / BRK 都有公司專屬 workaround），維護成本指數成長。

§3

ParsedFiling Schema

Architecture

3.1 Discriminated union

兩種 Item kind 行為完全不同（structured 有 prelude/blocks、flat 整段 chunk），用 union 表達，以 kind literal field 作 discriminator：

型別kind必要 fields
StructuredItem"structured"item（normalized key e.g. "1a"）、title、prelude（可為空）、blocks（≥ 1 個 Block）
FlatItem"flat"item、title、text（整 Item content）
Block—heading、text
ParsedFiling—metadata、items: list[ParsedItem]（stub 已 filter）

schema 用 Pydantic 模型實作，細節（base class 選擇、constructor 簽章）屬 implementation。

3.2 Schema 不變決策

  • Stub items 在 parse 階段直接 drop（不留 flag 給下游）
  • 無 markdown_content 字段（A 路線完全消除 markdown 中介）
  • FilingMetadata.converter 欄位移除（沒語義）
  • RawFiling dataclass 移除（fetch_filing_obj 直接給 TenK）

4.3 Noise filter rules

Rules 出處 · 以下五類 noise 來自 research_markdown_h3_cross_sector.md + research_prelude_block_relationship.md 跨 10 家公司、共 30 probes 的實證統計 — 不是先驗推測，是看到實際 H3/H4 list 後歸納的 pattern。

Financial-statement chapter dividerCONSOLIDATED STATEMENTS OF OPERATIONS, NOTES TO CONSOLIDATED FINANCIAL STATEMENTSregex ^CONSOLIDATED STATEMENTS OF .+ + 重複次數

4.4 Anchored search 規則

每個 noise-filtered H3 / H4 候選 heading，必須在 item_text 中以 standalone line（整行只有 heading 文字、不含 inline 提及）出現才算 anchored。Cross-Item collision 自動隔離 — Item 1 的 Overview 不會出現在 Item 7 的 plain text 中，因 plain text 是 Item-bounded。

4.5 Text fallback path 細則

當 markdown H3/H4 兩個 level 都無 anchored heading 時，走 Title-Case standalone-line detection：候選 line 必須符合 length window、Title-Case、不含 digit cluster / table separators、上下行 position signals（上行非句尾、下行為 long prose）、不是 Item 自身的 self-reference。具體 threshold 屬 implementation 細節。Research 30 probes 估計 3/30 會走 fallback（ADSK 1A、VZ 1A、XOM 1）。

4.6 is_stub_section v2 擴展

"... appears on pages N–M"JPM Item 7

§5

Chunking & Prelude payload metadata

Architecture

5.1 Chunkable segments interface

Parser 端對外提供一個 helper：給 ParsedItem 回傳一連串 chunkable unit，每個 unit 帶 (block_heading, text, item_prelude) 三個欄位。

  • FlatItem → 一個 unit，block_heading=None、item_prelude=None
  • StructuredItem → N 個 unit（一個 block 一個），每個 unit 都帶相同的 item_prelude

下游 chunking pipeline 對每個 unit 跑 splitter（見 §5.3），把得到的 chunk 直接 emit 進 Qdrant payload（schema 見 §6），chunk_index 是全 filing 範圍 counter。

5.2 Prelude as payload metadata（不獨立 chunk / embed）

為什麼選 payload metadata 而不是獨立 chunk？ 替代方案 — Prelude as separate chunks：prelude 像普通段落一樣 chunked + embedded，retrieve 時若 hit 到 block chunk，再額外 Qdrant query（filter same item, block_heading=None）拉回 prelude chunks 拼進 context。Reject 因為：(1) retrieve-time 多一次 Qdrant query（latency + N+1 query 模式）；(2) prelude chunk 自身有 embedding 會跟 block chunk 在 semantic 上搶 retrieval slot（稀釋 ranking）；(3) retrieve 端要靠 block_heading=None flag 區別 prelude 與一般 chunk，邏輯複雜易錯。本設計把 prelude 完全移到 payload metadata 端：不切、不 embed、payload self-contains，retrieve-time 零 augment query。
核心 insight · prelude（如 MD&A 的 scoping + FLS disclaimer + cross-reference）對 Item 內所有 block 都是 cross-cutting context。

Schema：StructuredItem.prelude 只在 ParsedFiling 結構中存一份，每個 block-derived chunk 在 Qdrant payload 中整段帶 prelude：

行為說明
Prelude 是否獨立 chunk + embed❌ No
Prelude 是否進 vector store 自身❌ No (沒 embedding)
Prelude 是否在 block chunk payload 帶✅ Yes (整段不切)
Retrieve 時 augment query❌ 不需要（payload self-contains）
Prelude 整體性✅ 保留
Storage overheadPrelude 重複存（per Item N block chunks 次），實際 100-300 MB 跨 corpus，可接受

LLM context preparation（retrieve-time 階段，不是 ingest-time）：

時序 · 使用者 query → vector match top-K chunks → load matched chunks payloads（payload 自帶 prelude）→ retrieve 端把 payload 組成給 LLM 看的 context（Item header + block heading + prelude + chunk text）→ 送進 LLM prompt。

零 augment query、零 retrieve-time 補拼邏輯 — payload 本身已 self-contain，retrieve 端只是格式化。

§7

Filing Store

Architecture
Filing store 是什麼？ 一個 machine 端的本地 cache，存的是 fetch + detect 完之後的 ParsedFiling 物件序列化成 JSON。不是給人類讀的（人類讀走 §10 inspect helper），是給下次同 ticker/year 跑 pipeline 的程式自己讀，省一次 EDGAR fetch + detect。
不是 pipeline 一路做完就好嗎？為什麼還要存中間產物？

Pipeline 確實是一路做完（fetch → parse → chunk → embed → Qdrant upsert），filing store 是 fetch+parse 完之後 順手 save 的副產物。真正的好處在「下次同 ticker/year 還要做什麼」這些場景：
場景沒 filing store有 filing store
改 chunk 策略重 ingest（e.g. chunk_size 512→1024 跑 eval 對照）重做 fetch (HTTP, SEC rate limit) + parse + chunk + embed跳過 fetch+parse → load JSON → re-chunk → re-embed
Inspect helper review detection 對不對（§10）重做 fetch + parse 才能 renderload JSON → render markdown，秒回
Retriever JIT process restart 後第一次 query 同 filing重做 fetch + parse + chunk + embedload JSON → re-chunk + re-embed（省 HTTP fetch）
edgartools 升級想看 detection 變化重做 fetch + parse 才能 diffload 舊 JSON vs 新 parse 結果直接 diff
換句話說 — filing store 是 fetch+parse 階段的 cache，Qdrant 是 embedding 階段的 cache，平行兩段。兩個 cache 因為 invalidation 觸發條件不同：filing store 只在 SEC filing 改版/edgartools 升級時失效（罕見）；Qdrant 在 chunking/embedding model 改時失效（更常見）。如果只存 Qdrant，每次改 chunk 策略都要重 fetch SEC，浪費 HTTP quota 跟 latency。

7.1 格式：JSON

維度決定
路徑data/sec_text/{TICKER}/10-K/{YEAR}.json
內容ParsedFiling.model_dump_json(indent=2) — 完整 ParsedFiling 物件（見下方範例）
YAML frontmatter移除（換成 JSON 整檔）
Markdown body移除（schema 不再有 markdown_content）

JSON 結構直接 mirror §3.1 ParsedFiling schema（metadata + items: [StructuredItem | FlatItem]），方便 schema validated round-trip。

7.2 LocalFilingStore interface

Store 提供 standard CRUD-ish 介面：save(filing) / get(ticker, year) -> ParsedFiling / exists(...) / list_filings(...)。內部用 JSON 格式作為 storage backend；YAML frontmatter / markdown body 等舊版專用 helper 不引入。

7.3 Filing store vs §10 inspect helper（兩個用途不重疊）

維度Filing store（本 section）§10 inspect subcommand

7.4 不存 markdown 中介層的理由

  • A 路線完全消除 markdown 中介（JSON schema round-trip 取代）
  • edgartools @lru_cache 是 process-level，跨 process（CLI / batch / retriever JIT 各自 cold start）EDGAR re-fetch 風險高（見 §7 開頭 callout）
  • JIT 路徑現有的 filing_hit cache 訊號（observability 用）必須保留
  • Human inspect 走 §10 helper（從 JSON derive markdown，不需要存中介層）

10.3 inspect subcommand（on-demand）

產出 chunking 之前 的 ParsedFiling 人類可讀 view，用於驗證 detection 對不對（整份 10-K 的所有 items + prelude + blocks 顯式展開），讓「retrieval 出問題不出在 chunking 之前」這個判斷可被人工確認。

Behavior：load ParsedFiling（cache miss 即時 fetch + parse）→ render 成 inspect markdown → 寫到 inspect/{ticker}_{year}.md（default 位置，可用 --out PATH override）→ 印 file path 給 user 開檔。

Repo policy：.gitignore 加 inspect/，user 自己 housekeeping。

10.4 ParsedFiling.to_inspect_markdown() 輸出 schema

整份 10-K 所有 items 都 render（StructuredItem 跟 FlatItem），單一 Item 區段結構如下：

  • Filing metadata（ticker、year、filing date、items list）
  • Item heading + kind + char count + block count
  • StructuredItem 額外列 prelude（blockquote 標示）+ 各 block heading + block content
  • FlatItem 列整段 text；stub filtered 的 Item 標 (stub filtered)

一眼能 diff 的東西：Item list（vs SEC ToC）、prelude 範圍（vs first block 之前內容）、block heading 是否真 heading（排除 ToC / table cell 誤判）、block content 範圍（沒切錯 item）。

50 rows，cover 4 axis：

Axis為什麼重要

15.2.3 Annotation workflow

沿 §15.2.2 的 4 axis 跨 sector 採樣 ticker，由 LLM 生成 question + sentence-level snippet 候選，人工 review 確認 snippet 真在 ground-truth filing 出現、item 標對。

15.3 A/B Eval Flow

A/B experiment 落在獨立 sibling scenario backend/evals/scenarios/sec_retrieval_html_vs_text/（own dataset.csv + eval_spec.yaml），不污染 production sec_retrieval/ scenario。Scorer code 共用 backend/evals/scorers/sec_retrieval_scorer.py（util 不分 scenario）。Sunset 時整 sec_retrieval_html_vs_text/ directory rm -rf，production scenario 完全沒被觸碰。

四象限 failure case mining：

象限對策
snippet hit + item hit完美，無需 review
snippet miss + item hit章節對段落錯（chunking quality 主訊號 — prelude lost / wrong block boundary / chunk 太碎），必 review
snippet miss + item miss完全錯，review root cause（detection 完全 fail / 抓錯 ticker / ground-truth 標錯）
snippet hit + item miss不可能（snippet 在 expected item 內）

15.4 Sunset Trigger（deliverable-based）

不設 timeline。三個 deliverable 都完成才啟動 sunset PR：

  1. Quantitative A/B 對照表 — 兩 collection metric 對照（snippet recall@5/@10, MRR, MAP）
  2. Qualitative 4 象限 failure case manual review — 對「新贏 / 舊贏 / 兩邊皆失敗」case 歸納 root cause
  3. Retrospective doc — 寫 HTML pipeline 的 systematic 弱點 + 確認新版有 fix；如有 regression case，先 fix 新版再重跑 eval；最終確定「採取 improvement 方向」

Sunset PR 內容見 §8.2。
