第一展廳

Prompt Engineering

System prompt 管不住模型輸出的案例 — 語言、格式、措辭，以及 policy 本身設計錯誤。

PE-01已修復

中文查詢原文漏進 tool arguments — 「微軟最近新聞」直接丟給搜尋引擎

情境

使用者用中文提問，agent 把「微軟最近新聞」原封不動傳給 Tavily search，而不是轉成 "MSFT recent news"；最終回應的語言也會與使用者語言不符。失敗只能靠人工翻對話發現。

Root cause

Prompt 只寫了一句模糊的 "Always think and search in English" — 沒有明確規則、沒有範例、tool schema 欄位層級沒有 guardrail。

解法

三層防禦：(1) system prompt 明確規則 + before/after 範例；(2) tool schema 的 query 欄位描述直接標 (MUST be in English)；(3) 建 8-case eval suite 用 CJK-ratio 程式化斷言防 regression。

量化

Baseline 5/8 失敗 → 修復後 8/8 通過（3-run stability 95.8%）。這個 failure 直接催生了整個 eval framework — repo 的第一個 eval 因它而生。

PE-02已修復殘留 3 bug

Citation 格式漂移 — prompt 鎖不住輸出格式，Sources 區塊 render 失敗

情境

LLM 輸出的 reference definitions 不符 CommonMark，前端 extractor 抓不到 → Sources 區塊空白、引用原文直接上屏。實際觀察到 5 種漂移：bullet 前綴 - [1]: URL、「來源：」標題黏住第一條 definition、definition 前缺空行、無冒號的 [N] URL、中文回應的全形括號 【N】。

Root cause

Prompt 沒有把模型鎖死在單一格式；輸出格式在真實流量下自然漂移（全形括號在中文語境特別頻繁）— 前端 parser 卻只認理想格式。

解法

雙端夾擊：prompt 端 LINK FORMAT 規則收緊（強制 [N]: <url> "<title>"、禁全形括號、禁 References heading）；前端 normalizeRefDefs() 修復管線容錯，每條 regex 對應一種「實測觀察到的 failure mode」。

量化

無 citation-format compliance scorer（缺口）；目前只有 E2E test 鎖 happy path。修復自身留下 3 個殘留 bug：strip regex 會誤刪 [2024] Revenue grew... 這種正文行、jump-link anchor 永遠不可達、cursor sentinel 在 code block 內字面上屏。

PE-05已修復

內部 tool-call budget 被模型說成「我被 rate limit 了」

情境

Tool-call 預算用盡時，middleware 的 block 訊息被 LLM 轉述成「我遇到 rate limit」— 把內部 orchestration 限制與 SEC/Tavily 的真實 429 混為一談，向使用者提供錯誤的失敗原因。後續連鎖：backend 把訊息改寫成 "this is NOT a rate limit…" 澄清後，前端的 /rate limit/i regex 反而匹配到澄清句自己，render 出「Too many requests」— 與本意完全相反。

Root cause

錯誤語意以 free-form prose 跨越「模型 ↔ harness ↔ 前端」三層，每一層都在對字串做猜測式解讀；缺乏結構化的 errorClass。

解法

Prompt 明文規則（"Do NOT tell the user 'I hit a rate limit'"）+ backend 措辭改為 "NOT an external rate limit" 讓誤匹配子字串消失；typed errorClass 列為獨立 refactor（未落地）。

第二展廳

Context Engineering

餵進模型的 context 本身有問題 — 錯的公司、錯的年度、溢出的 section、被英文淹沒的語言指令。

CE-01已修復

讀完英文 10-K 之後，回應語言漂移成英文

情境

Agent 讀入數千字的英文 SEC filing section 後，in-context 的英文量壓過 system prompt 裡的 language policy，忽略「以使用者語言回答」改用英文回覆。System prompt 在 context 開頭，離生成點最遠 — 距離就是弱化。

解法

Language-directive sandwich：tool output 用 _LANGUAGE_DIRECTIVE_PRE/_POST 前後包夾 section 內容，把語言指令錨定在「英文內容即將淹沒 policy 的邊界上」（程式碼自述）。並利用 dict insertion order 確保 pre-directive 一定先於 content 出現。

CE-02機制已修同型 year-filter 仍 open

跨公司污染 — 問 AMD 供應鏈風險，retrieval 給了 7 篇 Intel + 3 篇 NVIDIA、0 篇 AMD

情境

Naive dense retrieval 下，中文 query 問 AMD 供應鏈風險，top-10 chunks 全是競爭對手的 10-K。實驗報告原文：「系統完美地誤導下游 LLM 用 Intel 和 NVIDIA 的供應鏈描述回答 AMD 的問題。」

Root cause

語意相似度在「同產業競爭對手」之間最強 — 正是最不能搞混的公司；小語料 ticker（AMD 僅佔 4.5% chunks）被語意鄰居淹沒；中文 query → 英文 chunks 的跨語言條件放大 entity mismatch。

解法

Three-layer contract：payload metadata + Qdrant KeywordIndex + tenant-aware HNSW，query 時強制 must=[ticker=X] filter。

量化

18 題 A/B：naive mean p@10 = 0.622（std 0.312，AMD 三題平均 0.13，最低 0.00）→ filter 後 p@10 = 1.00。Caveat：實驗假設 oracle ticker，router 從 query 抽 ticker 的準確率未被隔離測試；且 "year": None 時 year filter 被丟掉 — 跨年份污染這條同型 failure 在 main 上仍未修。

CE-03已修復

Section 溢出邊界 — 要 Item 11，拿到的內容夾帶 Items 12–14

情境

get_section("11") 回傳的文字實際包含 Items 12/13/14 的內容 — agent 說「根據 Item 11」時，引用的出處對不上實際內容；char 數與 stub 判定也一起失真。Human UAT 抓到：AAPL FY2025 的 Item 11 有 2080 chars 未被標 stub，實為挾帶了後面三個 item 的 "incorporated by reference" 文字。

HE-03已修復 ×3

無效 ticker 的靜默垃圾 — 三個資料來源、三種「不報錯的錯」

情境

同一類 failure 在三代資料來源各發生一次。External API 對 invalid input 不 raise，回傳看起來像資料的垃圾 — harness 若原樣轉交，模型就把垃圾當事實。

三代標本
來源API 行為對 agent 的風險
yfinance .info回 {"currentPrice": null} 不 raise模型分不清「無效 ticker」與「休市」→ 幻覺解讀

解法

策展筆記

橫向 Patterns

  1. 模型相信 harness 給它的一切。靜默垃圾（HE-03）、錯誤分類（HE-04）、溢出的 section（CE-03）— 模型端看起來是「幻覺」或「答錯」，root cause 都在 tool 層。修模型行為之前先驗 tool 輸出。
  2. Prompt 硬化 + eval 固定是標準修法。PE-01 直接催生 eval framework；之後每個修好的失敗都留下 scorer、regression 斷言或 contract test。沒有 eval 的修復（PE-03）等於還沒修完。
  3. 格式管控要雙端夾擊。PE-02 同時收緊 prompt 與放寬前端 parser — 單靠 prompt 鎖格式在真實流量下必然漂移，下游一定要容錯。
  4. Policy 要貼合資料現實。PE-04 的教訓：對沒有 URL 的資料來源強制要求 URL，模型只有兩條路 — 抗命或捏造。政策設計錯誤會直接製造幻覺。
  5. Prompt 管不住規劃行為，harness 要兜底。HE-06 的 defense in depth：prompt 抑制 + single-flight 安全網。對模型的行為約束永遠要有非 prompt 的第二層。
  6. Human-in-the-loop 抓到自動化抓不到的。CE-03 的 section 溢出、HE-06 的重複呼叫都是 human UAT 發現 — 700+ 自動化測試對 concurrency 行為與真實資料內容缺陷都不可見。
