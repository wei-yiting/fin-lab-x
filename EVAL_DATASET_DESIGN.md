# FinLab-X Evaluation Dataset Design

**Date**: 2025-03-04
**Last Updated**: 2025-03-05
**Status**: Design Approved
**Scope**: Cross-version evaluation (V1-V5) for tracking response quality evolution

---

## 1. Overview

This document outlines the design for FinLab-X's evaluation dataset, a rubric-based evaluation system that spans V1 through V5 to track response quality improvements across architectural iterations.

### Design Principles

**Problem with topic-based categories**: The original 9 topic-based categories (market_data, sec_filing, etc.) cannot differentiate V1–V5 performance because most questions are answerable by V1's basic tools. Version discrimination requires categorizing questions by *what capability they demand*, not *what topic they cover*.

**Solution**: 6 intent × capability categories grounded in:
- **FinanceBench 4-type numerical taxonomy**: How to answer (Direct Lookup, Comparative Calc, Bivariate Calc, Multivariate Calc)
- **Compound-QA 5-type intent taxonomy**: What is being asked (Factual, Causal, Comparison, Hypothetical, Evaluative)

### Key Decisions

- **Dataset Size**: ~73 questions
- **Categories**: 6 intent-based (replacing 9 topic-based)
- **Scoring Method**: Binary yes/no checklist (15 checkpoints: 7U + 8Q), with Q7 as ternary (0/1/2); `score = (binary_sum + q7_score/2) / 15`
- **Generation Approach**: Parameterized LLM prompts — 6 prompt files, each invoked multiple times with `version_floor` + `calc_type` parameters
- **Reporting**: Score-only (no pass/fail thresholds), breakdowns by version_floor and discriminating_checkpoints

## 2. Dataset Schema

### 2.1 Question Format

```json
{
  "id": "FIN-FACTUAL-012",
  "question": "NVDA 2024 年各季度的毛利率是多少？",
  "category": "factual",
  "calc_type": "bivariate_calculation",
  "version_floor": "v3",
  "discriminating_checkpoints": ["U4", "U5", "Q2"],
  "ground_truth": {
    "type": "numeric",
    "formula": "gross_profit / revenue",
    "expected_value": "~73.7% (Q1), ~78.4% (Q2), ~74.6% (Q3), ~73.5% (Q4)",
    "evidence_source": "sql_query",
    "evidence_location": "financial_data table, NVDA, FY2024 quarterly"
  },
  "expected_tools": [
    { "name": "sql_query", "args": { "sql": "SELECT period, gross_profit/revenue FROM financial_data WHERE ticker='NVDA' AND date BETWEEN '2024-01-01' AND '2024-12-31'" } }
  ],
  "rubric": ["U1", "U2", "U4", "U5", "Q2", "Q5"],
  "tags": {
    "persona": "analyst intern",
    "multi_tool": false,
    "difficulty": "medium"
  },
  "notes": "Generated by factual prompt, version_floor=v3, calc_type=bivariate_calculation"
}
```

**Schema Fields:**

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | `FIN-{CATEGORY}-{SEQ}` (e.g., `FIN-FACTUAL-012`) |
| `question` | `string` | The question text (~50% English, ~50% Traditional Chinese) |
| `category` | `enum` | One of 6 intent categories (see Section 2.3) |
| `calc_type` | `enum \| null` | Numerical operation required (see Section 2.4) |
| `version_floor` | `"v1"–"v5" \| null` | Minimum version required to answer correctly (`null` for edge_case) |
| `discriminating_checkpoints` | `string[]` | Checkpoints expected to first pass at `version_floor` |
| `ground_truth` | `object \| null` | Expected answer for automated/judge verification |
| `expected_tools` | `object[]` | Expected tool calls with name and args |
| `rubric` | `string[]` | Applicable checkpoint IDs for this question |
| `tags` | `object` | Metadata labels |
| `notes` | `string` | Generation provenance |

**`ground_truth` sub-fields:**

| Sub-field | Type | Description |
|-----------|------|-------------|
| `type` | `enum` | `numeric` / `text_span` / `reasoning_checklist` |
| `formula` | `string \| null` | Calculation formula (for U5 verification) |
| `expected_value` | `string \| null` | Expected answer value or range |
| `evidence_source` | `string` | Tool that provides evidence (e.g., `sql_query`, `vector_rag_search`) |
| `evidence_location` | `string` | Where in the data source the answer lives |
| `required_elements` | `string[] \| null` | For `reasoning_checklist` type: list of elements a valid answer must contain |

**Edge case additional field:**

| Field | Type | Description |
|-------|------|-------------|
| `edge_type` | `enum` | `data_boundary` / `security_guardrail` / `out_of_scope` (edge_case category only) |

### 2.2 ID Naming Convention

- `FIN`: Fixed prefix for FinLab-X eval dataset
- `{CATEGORY}`: Uppercase category name (e.g., `FACTUAL`, `CAUSAL`, `COMPARISON`, `HYPOTHETICAL`, `EVALUATIVE`, `EDGE_CASE`)
- `{SEQ}`: 3-digit sequence number starting from 001, scoped per category

Example: `FIN-FACTUAL-012` = factual category, question #12

### 2.3 Categories (6 Intent-Based Categories)

| Category | Intent Type | Calc Types | Version Range | Count |
|---------|------------|-----------|--------------|-------|
| `factual` | Factual-Statement — "What is X?" / "How much is Y?" | `direct_lookup`, `bivariate_calculation`, `comparative_calculation` | V1–V3 | 23 |
| `causal` | Cause-and-Effect — "Why did X happen?" / "What caused Y?" | `null`, `relational` | V1, V4 | 10 |
| `comparison` | Comparison-and-Selection — "Which is better?" / "Compare X and Y" | `direct_lookup`, `comparative_calculation`, `relational` | V2–V4 | 13 |
| `hypothetical` | Hypothetical-Analysis — "What if X?" / "What would happen if Y?" | `relational`, `multivariate_calculation` | V4–V5 | 8 |
| `evaluative` | Evaluation-and-Suggestion — "Is X a good investment?" / "Should I...?" | `multivariate_calculation` | V5 | 8 |
| `edge_case` | Failure modes — data boundary, security guardrail, out-of-scope | `null` | All versions | 11 |

**Total: 73 questions**

**Why these categories discriminate versions:**

| Version | New Capability | Category/Calc Type That First Tests It |
|---------|---------------|---------------------------------------|
| V1 | yfinance + Tavily (direct tool call) | `factual/direct_lookup`, `causal/null` |
| V2 | Vector RAG over SEC filings | `factual/direct_lookup` (v2), `comparison/direct_lookup` |
| V3 | DuckDB SQL queries on structured data | `factual/bivariate_calculation`, `factual/comparative_calculation`, `comparison/comparative_calculation` |
| V4 | Graph query (supply chain relationships) | `causal/relational`, `comparison/relational`, `hypothetical/relational` |
| V5 | Multi-step Planner (orchestrate all sources) | `hypothetical/multivariate_calculation`, `evaluative/multivariate_calculation` |

### 2.4 `calc_type` Enum (FinanceBench-Derived)

| Value | Description | Version Required | Example |
|-------|-------------|-----------------|---------|
| `direct_lookup` | Single value from one tool call | V1 (yfinance/Tavily) or V2 (RAG) | "What is AAPL's P/E?" |
| `comparative_calculation` | Same variable across time periods or aggregation | V3 (SQL) | "AAPL's revenue CAGR 2020–2024" |
| `bivariate_calculation` | Two different metrics combined | V3 (SQL) | "NVDA's gross margin = gross_profit / revenue" |
| `multivariate_calculation` | Three or more metrics / cross-table join | V5 (Planner) | "Impact of China revenue loss on AAPL valuation" |
| `relational` | Graph traversal to find connected entities | V4 (Graph) | "Which companies depend on TSMC?" |
| `null` | No calculation required (news retrieval, refusal) | V1 or All | "Why did TSLA surge last month?" |

### 2.5 Tags

| Tag | Type | Description |
|-----|------|-------------|
| `persona` | string | Question asker role: `"retail investor"` / `"individual trader"` / `"analyst intern"` / `"sell-side analyst"` / `"portfolio manager"` |
| `multi_tool` | boolean | Requires 2+ tools to answer |
| `difficulty` | enum | `simple` / `medium` / `hard` |

## 3. Generation Pipeline

### 3.1 Parameterized Generation Approach

Each of the 6 categories has a single prompt file. The generation script calls each prompt multiple times with different `(version_floor, calc_type)` parameters. This ensures:
- Questions with the same intent but different capability requirements are stylistically consistent
- The capability constraint is precisely encoded in the question
- Questions are UNANSWERABLE by versions below `version_floor`

**Generation Matrix (13 invocations → 73 questions):**

| Category | version_floor | calc_type | Target Count |
|---------|--------------|-----------|-------------|
| `factual` | v1 | `direct_lookup` | 8 |
| `factual` | v2 | `direct_lookup` | 6 |
| `factual` | v3 | `bivariate_calculation` | 5 |
| `factual` | v3 | `comparative_calculation` | 4 |
| `causal` | v1 | `null` | 5 |
| `causal` | v4 | `relational` | 5 |
| `comparison` | v2 | `direct_lookup` | 4 |
| `comparison` | v3 | `comparative_calculation` | 5 |
| `comparison` | v4 | `relational` | 4 |
| `hypothetical` | v4 | `relational` | 4 |
| `hypothetical` | v5 | `multivariate_calculation` | 4 |
| `evaluative` | v5 | `multivariate_calculation` | 8 |
| `edge_case` | null | `null` | 11 |

**Version floor distribution:**

| Version Floor | Count | Categories |
|--------------|-------|-----------|
| V1 | 13 | factual (8) + causal (5) |
| V2 | 10 | factual (6) + comparison (4) |
| V3 | 14 | factual (9) + comparison (5) |
| V4 | 13 | causal (5) + comparison (4) + hypothetical (4) |
| V5 | 12 | hypothetical (4) + evaluative (8) |
| All | 11 | edge_case |
| **Total** | **73** | |

### 3.2 Prompt Structure

Each prompt file (`backend/evaluation/prompts/{category}_prompt.md`) contains:

1. **System Context Block** (injected from `system_context.md`): FinLab-X tool signatures by version, output schema, language requirements, prohibited patterns
2. **Category Intent Section**: The intent type, question asker personas, examples
3. **Parameterized Sub-sections**: One sub-section per `(version_floor, calc_type)` combination, each specifying:
   - Required capability
   - Evidence source (which tool)
   - Constraint (why lower versions fail)
   - Number to generate
   - Example questions
4. **Diversity Rules**: Ticker variety (≥10 per batch), ~50% Chinese/50% English, difficulty distribution
5. **Output Format**: JSON array matching the schema with all required fields

### 3.3 Common System Context

Injected into all prompts:

```
## Tool Signatures by Version

### V1 tools (always available)
def yfinance_stock_quote(ticker: str) -> dict      # price, P/E, market cap, 52w high/low, volume
def tavily_financial_search(query: str, ticker: str) -> dict  # recent news, events
def sec_filing_retriever(ticker: str, doc_type: "10-K"|"10-Q") -> str  # raw text sections

### V2 additional capability
vector_rag_search(query: str, ticker: str, doc_type: str) -> list[Chunk]
# Returns semantically relevant chunks with source metadata (section, page)
# Key capability: finds specific passages in dense SEC filings

### V3 additional capability
sql_query(sql: str) -> DataFrame
# DuckDB table: financial_data(ticker, date, period, metric, value, unit)
# Contains: quarterly revenue, gross_profit, operating_income, net_income, shares_outstanding
# Key capability: precise numerical computation on historical structured data

### V4 additional capability
graph_query(cypher: str) -> list[Node|Relationship]
# Graph: Company, Product nodes; SUPPLIER, COMPETES, CUSTOMER edges
# Populated from SEC "Business" sections
# Key capability: traverses company relationships not expressible in single filings

### V5 additional capability
# Orchestrator acts as Planner — invokes any combination of the above in multi-step sequence
# Key capability: synthesizes evidence from multiple data sources into coherent analysis

## Output rules
- ~50% English questions, ~50% Traditional Chinese questions
- Use specific, real US-listed tickers (AAPL, TSLA, NVDA, MSFT, GOOGL, AMZN, META, AMD, COIN, etc.)
- Include colloquial/Chinese company names requiring ticker resolution (臉書→META, 谷歌→GOOGL, 英偉達→NVDA)
- PROHIBITED: questions answerable from LLM parametric memory without tool access
```

### 3.4 Edge Case Sub-types

`edge_case` is subdivided by `edge_type` to test three distinct failure modes:

| edge_type | Count | Intent | Expected Behavior |
|-----------|-------|--------|------------------|
| `data_boundary` | 4 | Data doesn't exist or capability is insufficient | Refuse gracefully, explain limitation, no hallucination |
| `security_guardrail` | 4 | Malicious requests, prompt injection, manipulation | Refuse to comply, no tool invocation, no system info leak |
| `out_of_scope` | 3 | Non-financial or non-US-securities questions | Explain scope, redirect, no forced financial framing |

For all `edge_case` questions:
- `rubric` contains only `["U6", "U7"]`
- `ground_truth.type = "reasoning_checklist"`
- U1 is expected to be `False` (no tool call should occur)

### 3.5 Merge & Validation Pipeline

```
Stage 1: Generate (parameterized LLM calls per matrix row)
    ↓
Stage 2: Deduplication (question text similarity check)
    ↓
Stage 3: Schema Validation (required fields, enum values)
    ↓
Stage 4: Version Distribution Check (min 10 questions per version floor)
    ↓
Stage 5: Discriminating Checkpoint Coverage Check
          (each discriminating checkpoint must appear in ≥5 questions)
    ↓
Stage 6: Final Master Dataset
```

**Validation Checks:**
- ID uniqueness
- Schema compliance (all required fields present and valid)
- `version_floor` present on all non-edge_case questions
- `discriminating_checkpoints` non-empty on all non-edge_case questions
- Category distribution within expected ranges

## 4. Scoring Pipeline

### 4.1 Checkpoint Definitions (15 Total)

**U-Group: Correctness & Faithfulness (7 checkpoints)**

| ID | Name | Method | Description | Version Discriminating |
|----|------|--------|-------------|----------------------|
| U1 | `skill_invoked` | Programmatic | `len(tool_calls) > 0` 或 skill trigger 被觸發 | — (sanity check) |
| U2 | `correct_action_taken` | Programmatic | 選擇了正確的 tool/skill/路由，且參數正確（合併原 U2+U3） | V2, V3, V4 |
| U3 | `grounded_response` | LLM-as-judge | 所有 claims 都有 tool output 支持（Ragas Faithfulness binary 版） | V2 |
| U4 | `no_hallucinated_numbers` | Programmatic | 答案中的數字出現在 tool output 中 | V3 |
| U5 | `calculation_correct` | Programmatic | 基於 tool output 的計算邏輯正確（公式與結果皆驗證） | V3 |
| U6 | `appropriate_degradation` | Programmatic | 無可用資料或能力不足時，拒絕/說明限制，而非 hallucinate | All |
| U7 | `question_answered` | LLM-as-judge | Response 直接回答了被問的問題 | V4 (graph questions) |

**Q-Group: Quality (8 checkpoints)**

| ID | Name | Method | Version Discriminating | 說明 |
|----|------|--------|----------------------|------|
| Q1 | `source_attributed` | LLM-as-judge | V2 | 引用具體 source（文件名稱/段落），不只說「根據資料」 |
| Q2 | `specific_numbers_used` | LLM-as-judge | V3 | 有資料支持時使用精確數字，而非模糊描述 |
| Q3 | `no_knowledge_gap_hedging` | LLM-as-judge | V2 | 有資料支撐時不用「可能」「應該」等知識缺口語氣 |
| Q4 | `appropriate_uncertainty` | LLM-as-judge | All | 面對預測/未來/主觀問題時有適當免責聲明 |
| Q5 | `data_recency_explicit` | LLM-as-judge | All | 明確標示資料時效（如「截至 2024 Q3」），不只說「最新」 |
| Q6 | `multi_source_integrated` | LLM-as-judge | V4 | 整合了多個 tool/source 的資訊 |
| Q7 | `traceable_reasoning` | LLM-as-judge | **TERNARY (0/1/2)** | 結論有明確指向引用資料，推理鏈可追溯 |
| Q8 | `all_parts_addressed` | LLM-as-judge | V5 | 複合型問題的所有子問題都被回答到 |

**Q7 ternary scoring:**

| Score | 意義 |
|-------|------|
| 0 | 無推理，或結論完全無法追溯到引用資料 |
| 1 | 有推理但不完整，部分結論未連結到資料來源 |
| 2 | 完整可追溯，所有結論明確指向引用資料 |

**Removed checkpoints (與版本進步無關):**

| 原 ID | 名稱 | 移除原因 |
|-------|------|---------|
| Q3 (old) | `structured_presentation` | LLM 本身的 formatting 能力，不隨架構版本改變 |
| Q5 (old) | `no_unnecessary_hedging` | 拆分為 Q3（知識缺口 hedging）與 Q4（適當不確定性）|
| Q9 (old) | `professional_financial_language` | LLM 固有品質，無版本鑑別力 |

### 4.2 Rubric Selection Per Question

Not all 15 checkpoints apply to every question. The `rubric` field lists applicable checkpoint IDs.

**Always include:**
- `U1` (unless edge_case where no tool call is expected)
- `U6` (all questions)
- `U7` (all questions)

**Conditional inclusion:**
- `U2`: Include when `expected_tools` is non-empty
- `U3`: Include for V2+ questions (RAG grounding required)
- `U4`: Include when answer contains numbers from tool output
- `U5`: Include when `ground_truth.formula` is specified
- `Q1`: Include for V2+ questions (source attribution required)
- `Q2`: Include for V3+ quantitative questions
- `Q3`: Include when data-supported claims should be stated confidently
- `Q4`: Include for evaluative/hypothetical/forward-looking questions
- `Q5`: Include for real-time or time-sensitive data questions
- `Q6`: Include for multi-tool questions (`multi_tool: true`)
- `Q7`: Include for all non-trivial reasoning questions
- `Q8`: Include for compound multi-part questions

### 4.3 Scoring Method

- **Binary Checkpoints (U1-U7, Q1-Q6, Q8)**: Each = 1 (pass) or 0 (fail)
- **Ternary Checkpoint (Q7)**: 0 / 1 / 2，除以 2 換算為 0–1 分
- **Per-Question Score**: `score = (binary_sum + q7_score/2) / 15`
- **Aggregation**: Average/median across dataset, breakdowns by:
  - `category`
  - `version_floor`
  - `discriminating_checkpoints`
  - `difficulty`

### 4.4 Implementation

**Programmatic Checks (U1, U2, U4, U5, U6)**

Direct Python comparisons against tool call records and outputs:
- U1: `len(tool_calls) > 0`
- U2: Compare expected vs actual tool names AND args
- U4: Extract numbers from `final_answer`, verify each appears in any `tool_calls[].output`
- U5: Evaluate `ground_truth.formula` against tool output values, compare to `ground_truth.expected_value`
- U6: If any tool_call has error or empty output, check `final_answer` for graceful degradation language

**LLM-as-Judge Checks (U3, U7, Q1-Q8)**

One prompt per checkpoint, evaluated by a fixed judge model at `temperature=0`:
- Q7 prompt returns 0/1/2 (ternary) instead of YES/NO (binary)
- All judge outputs logged with input context for traceability
- Judge model and prompt versions locked per eval run

## 5. Reporting

### 5.1 Report Structure

**Overall Metrics**
- Average score across all questions
- Median score
- Score distribution histogram

**Breakdown Views**
- By `category`
- By `version_floor` (primary for V1–V5 comparison)
- By `discriminating_checkpoints` (shows which capabilities improved)
- By `difficulty`

### 5.2 Version Comparison

- V1 → V5 score trend (overall + by category)
- Discriminating checkpoint pass rate per version (shows when each capability was achieved)
- Q-group trend (quality evolution)
- U-group trend (correctness stability)

### 5.3 QA Sampling

- Each version update: sample 10-20% questions for manual review
- Purpose: Ensure LLM judge hasn't drifted
- Spot-check only, not full manual evaluation

## 6. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| LLM Judge Drift | Scoring inconsistency across versions | Fixed judge prompts + version locking + periodic spot checks |
| Tool Output Volatility | Real-time data changes affect U4 | Save tool outputs as evidence during eval execution |
| Version Floor Leakage | V1 agent partially answers V3 questions via hallucination | U4 + U5 checkpoints explicitly penalize this |
| Category Imbalance | Uneven version distribution skews comparison | Distribution check in merge pipeline (min 10 per version floor) |
| Graph Data Unavailability | V4 questions unanswerable if graph not populated | U6 (appropriate_degradation) distinguishes graceful refusal from wrong answer |
