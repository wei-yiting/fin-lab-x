03

CancelledError 排在 except Exception 之前,而且 cleanup「刻意」是 sync 的

高風險
backend/agent_engine/agents/base.py:505–516 · 518–544

為什麼挑這塊:cancellation 是這裡最難的 control flow。CancelledError 是 BaseException,所以必須排在 generic handler 之前,否則它會被誤報成 stream error。cleanup 刻意做成同步的,因為「sync code 不會被 CancelledError 中斷」。然後它再 re-raise。

except asyncio.CancelledError:
    # D35 abort cleanup. CancelledError is BaseException (not
    # Exception); list it before the generic except so it doesn't
    # collapse into the StreamError path.
    self._handle_abort_cleanup(mapper, handler)
    raise
except Exception as e:
    ...
    yield StreamError(error_text=sanitize_tool_error(str(e)))
問題 — 後果 / 沒走的那條路

如果這兩個 except block 的順序顛倒,使用者(以及 trace)會看到什麼?另外:為什麼 _handle_abort_cleanup 必須維持同步——如果它在 raise 之前 await 一個 Langfuse flush,具體會出什麼錯?如果你說不出最後那個 raise 在保護什麼,它還有在做事嗎?

05

用 chunk.id 的變化判 LLM-call 邊界,id=None 當作 continuation

高風險

為什麼挑這塊:整個 reasoning segment 的生命週期都靠這個 heuristic:當 chunk.id 改變就 flush segmenter、reset 它、再 arm 一個新的 reasoning_id;當 id is None 就把該 chunk 當成目前這個 call 的延續。這同時相依於三家 provider 的 chunk 形狀與順序。

if msg_chunk.id is not None and msg_chunk.id != self._current_llm_call_id:

    self._current_reasoning_id = None
