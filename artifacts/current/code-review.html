finnhub_client.py · get_finnhub_client

改了什麼

新檔。API key 在 call time 讀（不是 import time），讀不到就 raise ValueError("FINNHUB_API_KEY is not set.")，然後回傳新建的 finnhub.Client。

為什麼這樣設計

兩個目的：① lazy credential — 沒設 key 的環境 app 照常啟動，失敗縮小到「工具真的被呼叫的那一刻」，且訊息明確指出缺哪個 env var；② 這個函式就是整個測試體系的 patch seam。

實作細節

所有 unit test 都 patch("backend.agent_engine.tools.finnhub_client.get_finnhub_client") 回傳一個 MagicMock，把 .quote / .company_basic_financials 的 return_value 設好 — 不 patch finnhub.Client 本身，因為 SDK 的內部建構（session、headers）不是我們要驗的東西。missing-key 測試則反過來：不 patch seam + patch.dict(os.environ, {}, clear=True)，讓真實函式跑到 raise。

帶走的 外部 client 的建構式包成一個 module-level function，同時得到 lazy credential 解析跟乾淨的 mock seam — 比 patch SDK class 穩定得多。
§4

finnhub_client.py · fetch_quote / fetch_basic_financials

改了什麼

兩個 fetch 函式都對「合法 shape 的垃圾值」做判定：quote 的 c 與 pc 同時為 0/None → raise；basic financials 的 metric 為空 map → raise。錯誤訊息直接告訴 agent 可能原因（invalid / delisted / not covered by free tier）。

為什麼這樣設計

Finnhub free tier 對未知 symbol 不丟 error：/quote 回 {c:0, h:0, l:0, o:0, pc:0, d:None, dp:None}、/stock/metric 回 {metric: {}}。不攔的話，「AAPL 股價 $0」會一路流進 LLM context 變成幻覺素材。

實作細節

用 AND 而不是 OR：只檢查 c 會把「盤前合法為 0 的現價」誤殺 — c=0, pc>0（盤前）必須通過。in (0, None) 同時涵蓋「回 0」與「欄位缺失」兩種形態。(data or {}).get("metric") or {} 的雙重 fallback 讓 None 回傳值也走同一條 raise 路徑。WHY 註解直接寫在 guard 上方（review 修正項）。理論邊界：全新 IPO 第一天 pc=0 + 盤前 c=0 會誤判 — stake 不對稱（誤判 = agent 看到清楚錯誤可重試，不是錯資料進回答），接受。

帶走的 第三方 API 的「錯誤」不一定走 error channel — 合法 shape 的垃圾值（全 0、空 map）要在 client 層翻譯成明確 exception，別讓它默默流進 LLM 的 context。
§4

BASIC_FINANCIALS_CATALOG + FieldSpec

改了什麼

19 個 curated 欄位的單一對照表：output key → FieldSpec(metric_key, description)（NamedTuple）。涵蓋 52wk H/L、P/E、P/S、P/B、marketCap、beta、EPS、ROE、ROA、margins、流動性比率、D/E、股息率、成長率、均量。

為什麼這樣設計

Finnhub 的 metric key 是資料驅動字串（totalDebt/totalEquityQuarterly 有字面斜線、10DayAverageTradingVolume 數字開頭、Quarterly/TTM/Annual 字尾混用），沒有 SDK enum 背書，拼錯不報錯只會欄位靜默消失。集中一張表 = 拼字只有一個出錯點；tool 2（取值）與 tool 3（取描述）共用，保證 discovery 跟取值永遠一致。

實作細節

NamedTuple 而非 dict-of-dict：欄位存取有名字（spec.metric_key）、不可變、零額外依賴。描述字串會經 get_available_fields 原樣進 LLM context — 所以 marketCap 的描述從 "USD millions" 修成 "millions of reporting currency"（Finnhub 對 TSM 這類非美元 issuer 回的是當地幣別；review 抓到的真 bug）。19/19 key 已對真實 AAPL 回傳逐一驗證並固化成 live guard。

帶走的 「給 LLM 看的欄位白名單」本質是 contract surface — 用單一 source of truth + 真打 API 的 guard test 守住；描述字串也是 contract 的一部分，會被 LLM 當事實引用。
§5

finnhub_tools.py · 3 個 @tool

改了什麼

三個 LangChain @tool：finnhub_stock_quote（/quote 7 欄位映射）、finnhub_company_basic_financials（catalog 取值）、finnhub_get_available_fields（catalog 探索）。各自有 Pydantic input schema + InjectedToolCallId。

為什麼這樣設計

拆「報價 vs 基本面」是 rate-limit 預算決定（free tier 60 calls/min）：agent 只問價格時花 1 call。Present-only filter（in metric and is not None）不丟 null 給 LLM 省 token；tool 2 / tool 3 用同一個 filter 保證 agent 視角內部一致 — 不會「discovery 說有、取值拿不到」。

實作細節

共用模式：get_stream_writer() 包 try/except（非 streaming context 會 raise → 容錯成 None，工具照常運作）→ ticker.strip().upper() → 發 stream event（querying_stock / querying_financials / querying_fields 三種 status）→ 呼叫 client → 回 plain dict。錯誤一律 raise（不回 error dict），交給 _HandleToolErrors sanitize — 與 SEC tools 同一契約。quote 的 dayHigh/dayLow 是當日高低（h/l），52 週在 tool 2。兩個 catalog 迴圈刻意重複不抽 helper：output shape 不同（值 vs 描述），抽了只是 indirection。

帶走的 Agent tool 的切分粒度跟著外部 API 的計費 / 限流單位走，而不是「一個主題一個工具」的直覺。

news_search.py 改名 + __init__.py + v1–v5 configs

改了什麼

原 financial.py 淨刪 121 行（2 個 yfinance 工具 + import yfinance），只剩 tavily_financial_search → review 後改名 news_search.py（檔名跟著 purpose 走）；setup_tools() 改 import + register 3 個 finnhub 名稱；v1–v5 五個 yaml 的 tools list 從 2 項換成 3 項。

§1TL;DR

為什麼這個 PR 存在

Agent 的即時股價 / 市場資料一直走 yfinance — 非官方的 Yahoo Finance scraping， 沒有 API key、沒有 SLA，常踩 Yahoo 的 rate limit / IP throttling，tool call 隨機失敗。 本 PR 換成 Finnhub 官方 REST API（free tier，60 calls/min，有 key 有契約）， 走官方 finnhub-python SDK。順帶解掉兩個舊契約遺毒：prompt 強制附 Yahoo per-ticker URL（Finnhub 沒有對應頁面， 留著會逼 LLM 捏造連結）、以及 free tier 拿不到的 forward P/E。

Before — yfinance（非官方 scraping）
  • 2 個工具：yfinance_stock_quote（價格 + 52 週 + P/E 一包）、yfinance_get_available_fields
  • 無 API key、無官方契約 — Yahoo 改版 / 限流就直接壞

  • 契約含 forwardPE（Yahoo 有提供）
  • 無效 ticker 由 yfinance 內部行為決定（回 dict 缺 key）

  • 官方 finnhub-python>=2.4.28，free tier 60 calls/min，僅用 /quote + /stock/metric 兩個 free endpoint

7.1 marketCap 幣別描述錯誤 已修

問題：catalog 原本把 marketCapitalization 描述成 "Market capitalization (USD millions)" — 但 Finnhub 回的是 reporting currency millions。對 TSM（台幣計）這類非美元 issuer，錯誤幣別會經 finnhub_get_available_fields 的 description 直接漏進 agent 的回答，違反 zero-hallucination policy。Prompt 自己的 Example 2 就用 TSM — 這不是理論案例。

修正：描述改為 "Market capitalization (millions of reporting currency)"（commit b9fa675）。本輪 review 最有價值的發現。

7.2 all-zero 偵測的 AND 邊界 accepted

問題：c==0 AND pc==0 才判無效。理論邊界：全新 IPO 第一天（沒有 previous close）若 Finnhub 回 pc=0 且盤前 c=0，會被誤判成無效 ticker。

判定：接受。Stake 不對稱 — 誤判的後果是 agent 看到一條清楚的錯誤訊息（可重試、可改用其他工具），不是錯誤資料進回答；而把檢查放鬆成 OR 反而會放過真正的未知 symbol。Live test 對大盤股 + 無效 symbol 雙向驗證通過。若未來真踩到 IPO 案例，再針對性放寬。

7.3 present-only filter 的資訊損失 accepted

問題：tool 2 / tool 3 都用 metric[key] is not None 過濾 — 「Finnhub 明確回 null」與「該欄位不存在」對 agent 不可區分，是真實的資訊損失。

判定：接受。兩個工具用同一個 filter，agent 視角內部一致：不會發生「discovery 說有、取值拿不到」的矛盾。代價換來的是 token-efficient 契約（不丟一排 null 給 LLM）。若未來要區分，成本最低的位置是讓 tool 3 對 None 欄位回 {"available": true, "value": null}。

7.4 FinnhubAPIException 原樣 bubble accepted

問題：429（rate limit）/ 401（bad key）/ 403（premium 端點）都不在 client 層攔，直接 bubble 給 _HandleToolErrors middleware sanitize。SDK 有結構化的 e.status_code 可用但沒用。

判定：接受 — 與 SEC tools 的錯誤契約一致，sanitizer 會把 key 遮掉。Status-code 映射（如「403 = 此欄位需 premium」）是 YAGNI，等真的需要更細的 agent 訊號再加。
