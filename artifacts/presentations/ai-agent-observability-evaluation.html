Chapter 1 · Why this matters

The black box problem

Traditional Software

Logic is in the code

  • Deterministic. Same input → same output.
  • Bugs are findable: read the code, set a breakpoint.
  • Coverage means something. Tests pass = it works.
Generative AI

Logic is hidden in billions of weights

  • Probabilistic. Same prompt → different outputs.
  • Errors emerge from behavior — not stack traces.
  • Tests passing ≠ feature working. Need a different discipline.
The shift: In traditional software you debug code. In AI you debug behavior — and behavior only exists in observed outputs.

Chapter 1 · Why this matters

High-level metrics are not enough

A single accuracy number can tell you that something is wrong, but it rarely tells you what to fix.

What the metric hides
The failure path
The agent may have taken the wrong route through tools, prompts, or context long before the answer reached the user.
Why teams get stuck
Blind debugging
Without traces and structured review, improvements become guesswork, not engineering — and yesterday's errors reappear with every change.
What we actually need
Failure-mode visibility
We need to know the repeatable ways the system falls short so we can prioritize fixes rationally — by impact, not by intuition.
The shift in question: From "what's our accuracy?" → "which failure modes are dragging us down, and where in the trace do they originate?"
Chapter 1 · Why this matters

The discipline gap between teams

Less experienced

Blindly building

  • Try a new prompt. "Looks better?" Ship it.
  • Swap models. Hope the next one is smarter.
  • No quantified signal. Just vibes.
  • Same errors reappear after every change.
Experienced

Analysis-driven building

  • Look at outputs first. Categorize the failures.
  • Build the smallest eval that proves the error exists.
  • Each change is targeted, measured, repeatable.
  • Regressions caught before users see them.
Don't spend weeks theorizing about how to build the system. Build a quick one and look at it. — the most reliable shortcut in agentic AI development

Chapter 2 · Tracing

What is a trace?

A trace is the full lifecycle of one interaction — not just the final answer. It is the minimum unit of truth for debugging agents.

INPUT
User query and metadata.
→
PROMPTING
System prompt, examples, prompt version.
→
REASONING
Planning and intermediate model decisions.
↩
TOOLS
Search, retrieval, MCP calls, SQL, APIs.
→
OUTPUT
Final response and citations.
→
FEEDBACK
Thumbs, copy events, human review, labels.
Why traces matter
Without this chain, you only know that the answer disappointed the user. With it, you can see where logic, retrieval, or orchestration started to diverge.
Trace literacy
If you cannot inspect a single run end-to-end, you do not yet have an engineerable agent system.
Chapter 2 · Tracing

Tracing — anatomy of an agent run

Trace

The complete record of one agent execution — every input, every intermediate step, every output, with timing and cost.

Span

One unit of work inside the trace — an LLM call, a tool invocation, a retrieval, a sub-agent. Spans nest into a tree.

A trace is the flight recorder. Without it, you're guessing why the plane crashed.
Chapter 2 · Tracing

What hides inside a trace tree?

The final answer is often only the symptom. The root cause usually appears earlier in the execution tree.

Example failure cascade

USER QUESTION
Explain the impact and provide supporting evidence.
PLAN →
★ SEARCH SPAN — first failure
Bad retrieval result returns shallow or irrelevant evidence.
DOWNSTREAM EFFECT →
SYNTHESIS SPAN
Model fills gaps with unsupported claims because the context is weak.
VISIBLE SYMPTOM →
FINAL ANSWER
The response sounds fluent but is not trustworthy.
★ Fix the first failure Don't over-index on the final hallucination if a bad tool result created it. The cascade only ends when the root span is fixed.

What a trace tree gives you

  • Span-level attribution — which step failed?
  • Root-cause analysis instead of symptom chasing
  • A path from debugging to formal evals

The iterative development cycle

Strong agent engineering starts with rapid iteration — not long speculative design cycles.

Core claim: The fastest path to reliability is usually build → inspect → focus → improve — not weeks of guessing how the system might behave.
Chapter 3 · Build → Look → Improve

The decision moment: fix it, or measure it?

Path A

You see the error → you know the fix

  • Just fix it. Don't over-engineer.
  • Single-line prompt tweak, missing tool description, wrong temperature.
  • Re-run the same 10–20 cases to confirm.
Resist building infrastructure for a problem you can solve in 5 minutes.
Path B

The fix is unclear or will take many iterations

  • Stop. Build an eval first.
  • Without a quantified signal, every change is a guess.
  • The eval becomes your compass for the rest of the work.
Evals turn "feels better" into "measurably better."
Heuristic: If you'd need to re-test the same case more than 3 times to know if a change helped → build the eval.

  • FINAL ANSWER ✗ — the visible symptom, but the cascade started upstream.

Chapter 4 · Evaluation Design Concept

The 2×2 evaluation design matrix

Chapter 4 · Evaluation Design Concept

Observability vs evaluation

Seeing what happened is not the same as measuring whether the outcome was good.

Observability

What happened inside the run?

  • What input came in?
  • Which model and prompt were used?
  • Which tools were called, in what order?
  • How long did it take and what did it cost?
Evaluation

Was the outcome good enough?

  • Was the answer correct enough for the use case?
  • Did the agent choose the right path?
  • Did it satisfy the business constraint or rubric?
  • Did this release get better or worse?
You need both. Observability gives you explanatory visibility. Evaluation gives you quality judgment. Dashboards without evals leave you guessing if quality changed; evals without traces leave you guessing why.
Chapter 4 · Evaluation Design Concept

Three layers of evaluation

Chapter 4 · Evaluation Design Concept

When to use which evaluation layer

Layer What it measures Strength Weakness Use when…
End-to-End The final answer / artifact the user actually receives. Survives refactors. Closest to user value. Black box. Right answer can hide a bad path. Slow. Always. This is your North Star.
Trajectory Did the agent take the expected path of tool calls? Catches inefficient or unsafe routes. Tied to your current tool set. Rewrite when you change tools. Multi-step agents where how matters (cost, safety, audit).
Component One specific component, given a known input. Fast, cheap, low-noise. Tight feedback loop. Passing components don't guarantee a passing system. You've identified the bottleneck and want to iterate fast.
Pair them. End-to-end is your acceptance test. Component-level is your unit test. You need both — for the same reasons traditional software does.
Why component-level accelerates you: Cuts noise from the rest of the pipeline. Lets different teams own different metrics. Iterations get 10× cheaper.

Recommended evaluation design process
Knowing the eval shapes is not enough — you still need a process for choosing which ones to build. A concrete workflow: manual error analysis, failure modes, then automation.
Chapter 5 · Recommended Evaluation Design Process

Manual error analysis comes before automated metrics

50–100
traces · the sweet spot

Big enough to reveal patterns. Small enough that one engineer can read them all in an afternoon.

Why before automation?

  • You can't write an LLM-as-a-judge prompt for a failure mode you haven't named yet.
  • Dashboards show symptoms; manual reading reveals root causes.
  • Hand-labeled traces become the gold standard your future evaluators are calibrated against.
Insight: Understanding your failure modes — the specific, repeatable ways your system falls short — is infinitely more valuable than a generic accuracy score.
Production traces
If the app is live, sample 50–100 across different user intents. Don't cherry-pick.
Synthetic data
No users yet? Generate 50–100 realistic personas + queries. Imperfect > nothing.
Diversity matters
Cover your top intents. Otherwise you fix edge cases while the silent majority breaks.
Chapter 5 · Recommended Evaluation Design Process

The 5-step error analysis workflow

From qualitative → quantitative
The endgame is a frequency chart of failure modes you can sort by. That sort order is your roadmap.
An LLM can help cluster
After open coding, paste your notes into ChatGPT and ask it to propose 5–7 categories. Then revise.
Pareto applies
Usually 2–3 failure modes account for 60–80% of bad outputs. Fix those first.
Chapter 5 · Recommended Evaluation Design Process

Two techniques that change everything

Technique 1

First Pass — binary triage

Before you analyze anything deeply, label every trace just Good or Bad.

  • Filters out the noise immediately
  • Tells you your overall pass rate in one number
  • Focuses the deep dive on the 20–30% that actually fail
Don't try to invent rich labels on pass 1. Sort first, classify second.
Technique 2

First Failure Rule

When a chain breaks, only document the first thing that went wrong. Everything downstream is a symptom.

Fix the source, the cascade disappears. Tag downstream symptoms and you'll triple-count one root error.

Chapter 6
Component-level thinking
An agent is a system of components. The art is figuring out which one is dragging the system down — and how to improve only that one.
Chapter 6 · Component-Level Thinking

The component attribution principle

A research agent is failing. Where is it failing? Tag every span as good or subpar — but only relative to the input it received.

Attribution rule: A component is judged on the quality of its input. If summarization gets garbage docs and produces a garbage summary, that's not its fault. Don't penalize a component for upstream failure.
Chapter 6 · Component-Level Thinking

Improvement playbook — once you know which component

Non-LLM components

Search, retrieval, parsers, classifiers

  • Tune parameters. Top-k, similarity threshold, chunk size, date range, recency boost.
  • Replace the component. Try a different search engine, a different embedding model, a different RAG provider.
  • Pre/post-process. Query rewriting, filter low-score hits, deduplicate.
Often a 30-minute parameter sweep beats a week of prompt engineering downstream.
LLM components

In escalating order of cost

  1. Improve the prompt. Sharper instructions, few-shot examples, output format spec. CHEAPEST
  2. Try a different model. Stronger / cheaper / faster — match the model to the task.
  3. Decompose the task. Split one hard call into two simpler calls (generate → reflect / critique).
  4. Fine-tune. Last resort. Worth it only when you've hit 90–95% and need the last few points. HIGHEST ROI BAR
Stop-rule: Don't reach for fine-tuning when you haven't tried decomposition. Don't try a new model when you haven't tried 3 prompt variants. Walk the ladder.
Chapter 6 · Component-Level Thinking

Building intuition for which model fits which job

Play with new models often
When a model drops, give it 5 of your hardest tasks. Build a personal feel for its strengths and quirks before you ever wire it into a system.
Read other people's prompts
Open-source agent repos are a goldmine. Open the source, find the system prompt, see how senior practitioners actually talk to models.
Assess in your own workflow
Generic benchmarks don't predict your task. Run candidate models through your component-level eval. Decide on cost / latency / quality together.

Worked example — PII removal

Model classBehavior on a multi-step PII-strip prompt
Smaller open model (e.g. 8B-class) Answers simple questions fine. Misses one of three names. Adds an extra trailing list. Doesn't honor the output format.
Frontier model Follows the multi-step instructions reliably. Catches all PII. Output schema clean.

→ The "right" model depends on instruction-following pressure, not parameter count. Build the intuition firsthand.

Custom judge — what to encode

# dataset item { "input": { "question": "How do I trace a Python app?" }, "expected_output": { "response_facts": [ "Use the @observe() decorator", "Many integrations: LangChain, LlamaIndex, …" ], "trajectory": ["get_overview", "search_docs"], "search_term": "Python tracing" } } # judge prompt structure SYSTEM: You are a strict grader. RUBRIC: - For each fact in expected_output, decide if the agent's answer covers it. - Score = covered_facts / total_facts. RETURN: {"score": 0–1, "reasoning": "..."}
Calibrate the judge. Run the judge on your hand-labeled traces. If it disagrees with you > 20% of the time, your judge prompt is the error — not the agent.
Chapter 7 · Tooling Landscape

Datasets & annotation queues — keeping humans in the loop

Smart queue routing
Auto-route low-confidence outputs, user 👎, or judge-score < 0.5 into the queue. Don't ask humans to review easy cases.
Score configs
Pre-define the labels reviewers can apply (failure mode, severity). Consistency > freedom.
Promotion gate
Don't auto-promote everything. Review and curate before adding to golden dataset.

The continuous evaluation loop

The loop closes here. Production failures become offline test cases. Every regression you catch in prod is one you'll never ship again.
Chapter 8 · Continuous Loop & Production

Cost & latency — a good problem to have

Optimize quality first. When quality is good enough that users actually use the thing, cost and latency become the next frontier — and that's a sign the product is working.

Latency

  1. Benchmark first. Measure per-span latency. Find the bottleneck before guessing.
  2. Parallelize. Run independent tool calls concurrently — agent code is often serial out of habit.
  3. Right-size the model. Use a small/fast model for routing & classification, frontier model only for the hard step.
  4. Stream + cache. Stream tokens to the user. Cache repeated context (system prompts, RAG chunks).

Cost

  1. Cost per span, per intent. Find the heavy hitters. Often one component eats > 50% of cost.
  2. Prompt diet. Token-count your system prompt & retrieved context. Most are 2–3× larger than they need to be.
  3. Caching tokens. Prompt caching turns repeated context into a fraction of the price.
  4. Cheaper API for cheaper steps. Routing, intent detection, summarization rarely need a frontier model.
Same discipline as quality. Trace it, attribute the cost/latency to a specific span, decide on the smallest change that moves the metric. Then re-measure.
Chapter 9 · Wrap-up

Key takeaways

1 · Build → Look → Improve, on repeat
Don't theorize. Build a quick prototype, look at outputs, then decide whether to fix directly or build an eval.
2 · Trace before you measure
Without traces you have no evidence. With them, every "the agent is bad" becomes "this specific span produced this specific failure."
3 · Manual error analysis defines what to automate
50–100 hand-read traces. Cluster into 5–7 failure modes. Now you know what your LLM-as-a-judge should grade.
4 · Attribute failure to a component
Tag every span as good/subpar relative to its input. Improve the component with the highest subpar rate, not the loudest one.
5 · Walk the improvement ladder
Prompt → model → decompose → fine-tune. Cheapest first. Don't skip rungs.
6 · Close the loop
Production failures → annotation queue → golden dataset → next experiment. The loop is the product.
Start manual, find the truth, then automate the rigor. — the fastest path to an AI app users actually trust
